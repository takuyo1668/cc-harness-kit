# ARCHITECTURE — この環境はどういう前提で動くか

このキットは「Claude Code をどう設定すれば精度が上がるか」を、いくつかの明確な設計思想に基づいて
組み立てています。ここを理解すると、自分でルールを足したり削ったりする判断ができるようになります。

---

## 前提となる5つの本質原則

（出典: Claude Code 開発リードの運用哲学。`harness/CLAUDE.md` に本文）

1. **Plan modeはデフォルト** — 設計判断を伴うタスクは、いきなり実装せず計画に合意してから着手する。
   良い計画が成果の大半を決める。単純修正（1ファイル・既知パターン・差分を1文で言える）だけ直接着手。
2. **検証ループが品質を2〜3倍にする** — Claude に「結果を見る手段」を必ず与える（テスト・dev server・
   スクリーンショット・ログ）。**検証手段が無いと Claude は推測する**。推測完了報告が品質低下の最大原因。
3. **知識はPRに埋め込む** — ミスしたら即 CLAUDE.md か memory に追記し、次回から繰り返さない。
4. **繰り返し作業はコマンド化・サブエージェント化** — 同じ作業を2回手でやったら仕組みにする。
5. **過度にカスタマイズしない** — 「これを消したら Claude が間違えるか?」が No のルールは足さない。
   少なく濃いルールが最強。

## 中核メカニズム

### 1. 実行主体（enforcement）のあるルールだけを信じる
「心がけ」は破られます。このキットのルールには必ず**実行主体**（それを機械的に強制する仕組み）が紐づきます。

| ルール | 実行主体 |
|---|---|
| 破壊コマンドを止める | `settings.json` の `deny` ＋ `hooks/pretool-guard.sh`（2層） |
| 秘密ファイルを読ませない | `deny` の `Read(.env)` 等 ＋ pretool-guard のコマンド経由読み出し検知 |
| 完了報告前に検証する | `verification-before-completion` スキル |
| 思考前に型を選ぶ | `logical-thinking` スキル |
| プロンプトを記録する | `hooks/prompt-logger.sh`（UserPromptSubmit） |
| memory肥大を検知する | `hooks/harness-health.sh`（SessionStart） |

→ 新しいルールを足すときも「実行主体は何か」を必ずセットで決める。持てないなら memory 止まりにする。

### 2. 事故防止の2層防御
復旧不可能な操作（`rm -rf` / `git push --force` / `git reset --hard` / `DROP TABLE` / `TRUNCATE` 等）は
2層で止めます。片方をすり抜けても、もう片方が止める。

- **第1層 `settings.json` の deny**: Claude Code 本体が発行前にブロック。
- **第2層 `hooks/pretool-guard.sh`**: Bash コマンドの中身を正規表現で検査し、deny を返す。
  `psql -c 'DROP TABLE'` のような**設定の deny をすり抜ける経由**も捕捉する。

> 限界の明示: この2層は「うっかり事故」の防止層です。悪意ある回避（変数展開・base64 エンコード等）は
> 正規表現では原理的に防げません。そこは Claude Code の permissions と権限モードが担います。

### 3. 検証ループ（推測完了報告の禁止）
最大の品質ドライバ。実装・修正・設定変更を「完了」と言う前に、`verification-before-completion` スキルの
「言い訳封じ表」を自己照合します。テスト・ビルド・動作確認の**証拠**が無い完了報告を禁止します。
フロントエンド変更なら「ヘッドレスChromeでスクショ→目視」まで、`rules/frontend.md` が要求します。

### 4. Generator-Evaluator 分離（高リスク時のみ）
本番デプロイ物・外部API連携・セキュリティ・新規設計など**高リスクなタスクに限って**、
生成役と評価役を分けます。評価基準は事前にチェックリスト化し、「良い」「適切」等の曖昧語を禁止して
検証可能な条件だけで判定します。低リスクなローカル変更には適用しません（オーバーヘッドの無駄を避ける）。

### 5. コンテキスト管理
長いタスク（7フェーズ以上）は各フェーズをサブエージェントに分け、短いコンテキストで動かして成果物だけを
引き継ぎます。7フェーズ未満は自動要約（compaction）に任せます。要約後も保持したい原則は
`context-essentials.md` に置きます。

### 6. サブエージェント編成
`agents/` に22種の専門役（architect / code-reviewer / devops / data-analyst / 各種マーケ専門家 等）を定義。
リサーチや並列可能な作業を委譲して速度を上げます。`/pm` コマンドはこの中からタスクに応じたチームを動的に編成します。

## フック4種の役割

| フック | 発火タイミング | 役割 |
|---|---|---|
| `pretool-guard.sh` | PreToolUse (Bash) | 破壊コマンド・秘密読み出し・秘密の外部送信をブロック |
| `prompt-logger.sh` | UserPromptSubmit | 全プロンプトを `~/.claude/prompt-log/YYYY-MM.jsonl` に記録（振り返りの一次データ） |
| `web-evaluator.sh` | PostToolUse (ブラウザ操作) | HTML/CSS変更後に視覚検証を促す（jq依存） |
| `harness-health.sh` | SessionStart | memory肥大を検知して1行警告 |

いずれも「記録・検査に失敗してもセッションを止めない（exit 0）」設計です。フックがユーザーの作業を
ブロックしてしまう事故を避けるためです（pretool-guard の意図的な deny を除く）。

## フックのトラブルシューティング

- **フックが全く動かない** → `settings.json` の `hooks` のパスが `$HOME/.claude/hooks/...` になっているか。
  `$HOME` が解決されているかを `echo $HOME` で確認。
- **pretool-guard が deny を返さない** → `python3` が入っているか（`python3 --version`）。
  pretool-guard は python3 が無いと安全側で deny する設計なので、逆に全部 deny されるなら python3 未導入。
- **web-evaluator でエラー** → `jq` 未導入。`brew install jq`。使わないなら該当フックを settings から外してよい。
- **フックを一時的に止めたい** → `settings.json` の該当フックの1ブロックを削除。ファイル自体は残してよい。

---

## 設計上の「あえてやっていない」こと（過度なカスタマイズの回避）
- グローバルに効くルールを盛りすぎない。領域別ルール（backend/frontend/infrastructure）は
  「全セッション常時読み込み」だからこそ総量を絞っている。
- 曖昧な「品質を上げる」系のルールは入れない。実行主体を持てるものだけ残す。
- 月1回「このルールはまだ必要か? 無くて品質が落ちた実績はあるか?」を問い、死んだルールを削る
  （`/weekly-review` に組み込み）。増やすより減らす方を意識する。
