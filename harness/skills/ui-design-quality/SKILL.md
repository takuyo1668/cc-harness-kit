# UI Design Quality — 品質3層定義 + Wave方式

大量画面のUIデザインを、品質を落とさずに制作するためのフレームワーク。
品質を3層に分けて定義し、Wave方式で制作することで、40画面以上でも軸がブレない。

## 基本原則

- 品質は一枚岩ではない。3層に分解して管理する
- すべての画面に最高品質を求めない。層ごとに投資を最適化する
- 作る順番は「1画面ずつ完成」ではなく「全画面を層ごとに一括」で進める

---

## Phase 0: 画面分類

デザイン作業を始める前に、全画面をL1/L2/L3に分類する。

### 分類基準

| 層 | 対象画面 | 画面数の目安 | 判断基準 |
|---|---|---|---|
| L1のみ | 設定、規約、確認ダイアログ、管理画面等 | 全体の50-60% | ユーザーが目的を果たせれば十分な画面 |
| L1 + L2 | ホーム、一覧、検索結果、詳細画面等 | 全体の30-40% | ユーザーが長く滞在する画面 |
| L1 + L2 + L3 | オンボーディング、初回体験、決定的エラー等 | 全体の5-10% | ブランド印象を決定づける画面 |

### 分類の出力フォーマット

```markdown
## 画面分類表

| # | 画面名 | 品質層 | 理由 |
|---|--------|--------|------|
| 1 | ログイン | L1 | 標準的な認証画面 |
| 2 | ホーム | L2 | 毎日使うメイン画面 |
| 3 | オンボーディング | L3 | 初回印象を決定づける |
```

**ユーザーと分類表を合意してからデザインに着手する。**

## L1：基礎品質（すべての画面で必須）

壊れていないこと。一貫していること。プロの仕事として最低限守る水準。

### L1 チェックリスト

- [ ] コンポーネントはデザインシステムライブラリから使用する（新規作成禁止）
- [ ] 色はセマンティックトークンを参照する（hex直打ち禁止）
- [ ] Auto Layoutで組む（固定高さはスクロールコンテナのみ例外）
- [ ] レイヤー名: `[Category]/[Component]/[Variant]/[State]`
- [ ] 5状態を定義する: Default / Hover / Focus / Error / Disabled
- [ ] テキストスタイルはデザインシステムのタイポグラフィトークンを使用
- [ ] アイコンはデザインシステムのアイコンセットから選択
- [ ] 最小タップ領域 44x44px を確保（モバイル）

### L1 の作業ルール

- AIが主担当。SKILL.md の基準に従って機械的に適用する
- 人間のレビューは抜き打ちチェックのみ
- 1画面あたりの目標時間: 短い（量を回すフェーズ）

---

## L2：体験品質（主要画面で実現）

使っていてストレスがないこと。心地よいこと。

### L2 チェックリスト

- [ ] 情報のヒエラルキーが明確で、視線が自然に流れる
- [ ] 余白・タイポグラフィのリズムが画面全体で統一されている
- [ ] 3状態が設計されている: Loading（スケルトン）/ Empty / Error
- [ ] 状態遷移がスムーズで、ユーザーが「今どこにいるか」を見失わない
- [ ] フォームのインライン バリデーションが設計されている（該当画面のみ）
- [ ] スクロール時の情報量が適切（1スクロールで得られる情報に意味がある）
- [ ] レスポンシブ: デスクトップ / タブレット / モバイルの3ブレークポイント

### L2 の作業ルール

- AIと人間の協業。人間が方向を示し、AIが実装する
- 例:「このLoadingは0.3秒後にスケルトンを表示して、データ取得完了後にフェードインさせる」→ AIが設計
- 人間は「何を」の判断に集中し、「どうやって」はAIに任せる

---

## L3：感動品質（特定画面に集中）

使った人が誰かに話したくなること。記憶に残る体験。

### L3 チェックリスト

- [ ] 予想を超える体験がある（「ここまでやるか」という驚き）
- [ ] ブランドの印象を決定づけるビジュアルインパクト
- [ ] マイクロインタラクションが意図的に設計されている
- [ ] エモーショナルデザインが組み込まれている（コピー、イラスト、動き）
- [ ] パフォーマンスを損なわない（アニメーションは60fps、LCPに影響しない）

### L3 の作業ルール

- 人間が主担当。AIは実装アシスト
- 何がユーザーの心を動かすかは人間が設計する
- L3を狙う画面は全体の5-10%に限定する。広げすぎない

## Wave方式 — 制作の進め方

全画面を波のように、層ごとに一括で進める。

```
Wave 1（L1）: 全画面のワイヤーフレームを一気に作る
  ↓ 全画面のL1品質をレビュー
Wave 2（L2）: 主要画面のビジュアルと体験を磨く
  ↓ L2対象画面をレビュー
Wave 3（L3）: 限定した画面に感動品質を集中投下する
  ↓ L3対象画面をレビュー
```

### Wave 1 の進め方

1. デザインシステムのコンポーネント・トークンを確認する
2. 全画面のL1チェックリストを満たすワイヤーフレームを作成する
3. 画面間の一貫性を確認する（ナビゲーション、共通パターン）
4. L1チェックリストで全画面を検証する

### Wave 2 の進め方

1. L2対象画面を優先順位付けする（ユーザーの利用頻度順）
2. 画面ごとに体験の重点ポイントを定義する
3. Loading / Empty / Error の各状態を設計する
4. レスポンシブ対応を実施する
5. L2チェックリストで対象画面を検証する

### Wave 3 の進め方

1. L3対象画面（2-4画面）を確定する
2. 各画面の「驚きポイント」を言語化する
3. マイクロインタラクション・エモーショナルデザインを設計する
4. パフォーマンス影響を確認する
5. L3チェックリストで対象画面を検証する

## Figma MCP 連携時の注意

Figma MCPでデザインを作成・取得する場合:

- `get_design_context` でコンポーネントライブラリの構成を最初に確認する
- 新規コンポーネントを作らず、既存ライブラリから選択する（L1基準）
- セマンティックトークン（色・タイポグラフィ）はFigmaの変数機能を使う
- レイヤー命名規則 `[Category]/[Component]/[Variant]/[State]` を厳守する

## アンチパターン

| やりがち | 正しいアプローチ |
|---|---|
| 全画面にL3を求める | L3は5-10%に限定。残りはL1/L2で十分 |
| 1画面ずつ完成させる | Wave方式で層ごとに一括進行 |
| 品質基準を曖昧にする | チェックリストで検証可能な条件に落とす |
| AIに「いい感じに」と指示 | L1はルールで制約、L2は具体的な体験指示 |
| L1を人間がレビューし続ける | L1はAI+抜き打ち。人間はL2/L3に集中 |
