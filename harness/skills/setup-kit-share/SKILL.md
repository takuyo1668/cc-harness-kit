# setup-kit-share — 環境ベストプラクティスの配布キット化

ローカルで完成した運用（フォルダ構成・スキル・コマンド・自動化・アプリ設定）を、**まっさらなPCの他人でも一度で再現できるキット**にして配る。

## キットの構成原則: 読者別に3+1ファイル

受け手ごとに入口を1つにする（非技術者はSETUPを開かず、AIはGUIDEを読まない。問い合わせコストが下がる）:

| ファイル | 読者 | 内容 |
|---|---|---|
| `GUIDE.md` | 人間 | これは何か／何が嬉しいか／仕組み／まっさらPCからのセットアップ手順／日常の使い方／FAQ（データの置き場所・費用・やめ方）／困ったら |
| `PROMPT.md` | 人間 | Claude Code に貼る数行の起動指示文＋キットの中身一覧 |
| `SETUP.md` | Claude | セットアップ作業指示書の本体。生成すべき全ファイルの本文を埋め込み、これ1枚で完結させる |
| `SLACK.md` | 配布者 | Slack告知文（`slack` スキルの選択式ワークフローで作る） |

## SETUP.md の書き方

- **手順0a: 前提ツール確認** — まっさらなマシン対応。Claude Code自身が動いている＝導入済みなので、それ以外（git・対象アプリ等）を存在チェックし、不足分は「人間向けの導入手順を提示して待つ」。AIが代行できない事実を明記する
- **手順0b: ヒアリング** — AskUserQuestion で設置パス等を1回だけ確認。**プレースホルダ（`{{VAULT_PATH}}` `{{HOME}}` `{{TODAY}}` 等）は手順0で全て定義**し、置換してはいけない例外（アプリ側のテンプレート変数など）も明記する
- **生成ファイルは全文埋め込み** — 「〜のようなファイルを作る」ではなく本文をコードブロックで埋め込み、「本文を改変せずプレースホルダ置換のみ」と指示する。再現の忠実度が指示の曖昧さで劣化しない
- **担当を層で分ける** — 「Claudeが生成する層」「人間がGUIで操作する層（チェックリストとして最後に提示）」を冒頭の表で宣言する
- **無人自動化（launchd/cron＋ヘッドレスAI）を含む場合、有効化コマンドは人間に提示するだけにする**。AIが無人実行の仕組みを自分で有効化しない（権限ゲートにも止められる）。解除コマンドと他OS代替も併記
- **手順最終節: 動作確認と完了報告** — テストデータ作成→検出/動作確認→掃除の許可を取る→完了報告フォーマット、まで指示書に含める
- 個人情報（氏名・社名・実パス・既存データ・トークン）は入れない。すべてプレースホルダ化

## ワークフロー

### 1. 対象の棚卸し

配布したい運用の構成要素を層で洗い出す: ①データ/フォルダ構造 ②アプリ側設定（GUI・プラグイン） ③Claude側（スキル/コマンド/CLAUDE.md/自動化） ④定期実行（launchd/cron）。git管理下にある正本ドキュメントがあるなら、キットにはその汎用化版を埋め込む。

### 2. キット生成

`~/<対象名>-kit/` に4ファイルを作成。GUIDEは非技術者向けの正確な日本語（比喩に頼らず、専門用語は初出で一言説明）。

### 3. E2E検証（必須・省略しない）

サブエージェントを「新環境のClaude Code役」として、**scratchpad内の疑似HOME**にSETUP.mdを実際に適用させる。検証観点:

1. SETUP.mdだけで迷いなく完走できたか（曖昧・不足箇所の列挙）
2. 生成物の構造一覧（find）
3. プレースホルダ置換漏れ（`grep -rn '{{'`）
4. 動作確認手順の正例・負例が通るか
5. 総合判定と修正推奨

指摘はSETUP.mdに反映してから配布する（レンダリング/実行を見ずに配布しない）。

### 4. Drive格納

1. Driveに `<対象名>-kit` フォルダを作成（`mcp__claude_ai_Google_Drive__create_file`、mimeType=folder）
2. 各ファイルを `textContent`＋`contentMimeType: text/markdown`＋`disableConversionToGoogleType: true` でアップロード（**Googleドキュメント変換させない**。受け手がDLしてClaudeにそのまま読ませるため）
3. 返却された `fileSize` をローカルの `wc -c` と突合し、バイト数一致を確認する
4. **共有設定は変更しない**（既定=本人のみ）。公開範囲の決定は人間に委ねると報告する

### 5. Slack告知文

`slack` スキルを起動して選択式で作成（冒頭テーマ→本文構成→末尾補足の順に選ばせる）。完成文は `SLACK.md` に保存し、`LANG=en_US.UTF-8 pbcopy` でクリップボードへ（pbpaste バイト数検証つき）。入手案内はDrive配布ならフォルダリンクに合わせる。

### 6. 完了報告

生成ファイル一覧／DriveフォルダURL／バイト数検証結果／人間に残る作業（共有設定・Slack投稿）を提示。

## Known Pitfalls

- Drive MCPへ大きいファイルを渡すときはbase64より `textContent` が効率的（UTF-8ならそのまま渡る）。アップロード前に実ファイルをReadし直して本文のドリフトを防ぐ
- SETUP.md内のコードブロック入れ子は外側を4連バッククォート（````）にする。インラインコード内にバッククォートを書くときは `` 二重バッククォート `` を使い、`\`` エスケープは使わない（生成物に残る）
- git初期化を含める場合: 空フォルダは `.gitkeep`、pushは `git branch -M main` を先に
