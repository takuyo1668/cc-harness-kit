# 日本語ライティング規範 — 違和感を消し、推進力を与える

出典: k16shikano（鹿野桂一郎・ラムダノート）の japanese-tech-writing / cognitive-rhythm-writing 規範（Unlicense）を土台に、AI生成日本語特有の違和感パターンを加えて一般文書向けに再構成したもの。

## 思想 — 違和感の正体は2層ある

AIが書いた日本語が「読めるのに、なんとなく嘘くさい」のは、別々の2つの問題が重なっているからである。

1. **文レベルの違和感（減点の層）**: LLM特有の空句（「重要なのは〜」「多角的に」）、翻訳調（「〜することができます」）、冗長な言い換え、論証の穴。1文ずつは小さな傷だが、蓄積すると「機械が書いた」印象になる。
2. **文章レベルの平坦さ（推進力の欠如）**: 全文が同じ認知モード（ずっと断定、ずっと解説）で書かれていて、緩急がない。減点を全部消しても、これが残ると「正しいのに読み進める気がしない」文章になる。

層1を消すだけでは足りず、層2を作るには層1が消えている必要がある。だから常に**層1→層2の順**で適用する。

- 層1の規則集: [references/dissonance.md](references/dissonance.md) — 書く前・推敲時に必ず読む
- 層2の規則集: [references/rhythm.md](references/rhythm.md) — 読み物として読ませたい文章（記事・解説・章）で読む。事務連絡・手順書・短文には適用しない

## 唯一の判定軸 — その文が更新するのは「状況」か「文書」か

迷ったらこれだけで判定する。

- **状況を更新する文**: 対象世界の出来事・データ・発言、または書き手の判断の状態（思い込み、保留、譲歩、告白）を新しく伝える。→ 残す。
- **文書を更新する文**: この記事・この節・ここまでの話が「どう見えるか」「次に何を書くか」だけを伝える。→ 原則削除する。

駄文の典型はすべて後者である。「本記事では〜を解説します」「ここまでで〜は見えた。次は〜を見る」「まとめると」「誤解しないでほしいのだが」。これらは論点を増やさず「ちゃんと書いている感」だけを付ける。

注意: 駄文は短い断定に整形しても駄文のままである。「テクニックの列挙はしない。」のように短くリズムが良い文ほど残りやすい。短さは残す理由にならない。話題が本文自身なら削る。

例外として残せる4つの形（詳細は rhythm.md）: ①具体的な誤読を「」で引用して退ける反論処理 ②問いの設置と回収 ③章・節の境界に置く読者への依頼 ④例の枠の開閉（「〜としよう」）。

## 大原則 — 装置は実現するものであって、宣言するものではない

この規範の語彙や操作（「緊張」「回収」「先に答えを半分だけ置く」「あえて短く言えば」）を本文に書かない。緩急も問いも、内容そのもので実現する。装置が正しく機能していれば、読者は装置の存在に気づかない。操作の宣言文は、それ自体が駄文である。

## ワークフロー

### 新規に書くとき

1. 読者と文書種別を特定する（誰が・何のために読むか。非技術者向けなら専門用語は初出で言い換える）
2. references/dissonance.md を読み、禁止パターンを頭に入れてから書く
3. 読み物系なら references/rhythm.md も読み、冒頭で未回収の緊張を1つ作る設計をしてから書く
4. 書き上げたら下の「執筆後の点検」を機械的に実行する

生成時の制約: 緩急・緊張の材料は状況側（対象の性質・データ・出来事・書き手の判断）からしか取らない。材料が見つからない位置は、無理に飾らず平坦なまま残す。

### 診断・リライトするとき

1. まず症状を特定する（下の症状→処方表）
2. 層1を先に掃除する（dissonance.md の禁止リストで機械的に検索・置換）
3. 読み物系なら層2を設計し直す（rhythm.md）
4. 元の文章の事実・主張を勝手に増減しない。不確実として書かれた文（「かもしれない」「だろう」)を機械的に断定へ変えない

## 執筆後の点検（毎回、この順で機械的に実行）

1. **話題テスト**: 段落頭の文と独立した短文をすべて拾い、「状況更新か文書更新か」を判定。文書側は例外4形に該当しない限り削除または状況側の文へ書き換え。推敲で新しく書いた文も書いた直後に再判定する（駄文の最大の混入経路）
2. **LLM口調の検索**: dissonance.md の禁止リスト（「重要なのは」「〜に他ならない」「非常に」「掘り下げる」「いかがでしたか」等）を検索し、機械的に除去
3. **翻訳調の検索**: 「することができ」「を行う」「〜において」「〜となっています」を検索し、直接的な動詞に置換
4. **文末の単調チェック**: 同じ文末（「〜です。〜です。〜です。」「〜だ。〜だ。」体言止め連発）が3連続していないか
5. **漏出テスト**: この規範の語彙（「緊張」「回収」「答えの半分」等）が本文に現れていないか。節末に「次は〜を見る」型の進行予告がないか
6. **緊張台帳**（読み物系のみ）: 立てた問い・思い込み・約束を列挙し、それぞれの回収位置を指す。指せないものは回収を書き足すか問いごと削る
7. **境界の点検**: 二人称の呼びかけ・依頼・謙抑が本文中盤に混ざっていないか。あれば冒頭・結びへ移すか削る

## 症状→処方

- **AIが書いたとすぐわかる**: 層1の掃除。dissonance.md の「LLM口調」「AI日本語の型」を検索置換
- **正しいのに読み進める気がしない**: 未回収の緊張がない。rhythm.md の冒頭の型で緊張を1つ作り、以降も常に1つは開いておく
- **全段落が同じ調子で疲れる**: 文の拍がない。rhythm.md「文の拍」— 短文で足場→長文で流す→短文で止める
- **緩い文はあるのに弛んで見える**: 緩みが進行実況になっている。話題テストにかけ、状況側の文（判断の保留・思い込み）へ書き換える
- **結びが説教くさい・「いかがでしたか」的**: 抽象論のまま閉じている。読者がすでに持っている具体（冒頭の場面・読者の経験）へ着地させ、未決の問いを1つ残して終える
- **冒頭が事務的**: 態度のない議題表になっている。削るのではなく予告文に態度を与えるか、予告の前に読者の実感・抵抗の処理を置く
- **箇条書きだらけで本文がない**: 箇条書きを段落に従属させる。列挙したら各項目を直前の具体へ着地させる
