# 完了報告前の検証ゲート（言い訳封じ）

## このスキルの目的

Claude最大の品質低下要因は「**推測完了報告**」＝検証していないのに「完了しました」「動くはず」と報告すること（CLAUDE.md 本質原則#2）。
このスキルは、完了と言う前に**自分の言い訳を表で叩く**ことで、推測報告を構造的に止める。

> **鉄則: "seems right"（たぶん合ってる）は完了ではない。**
> 証拠（テスト出力・ビルド結果・UIスクリーンショット・実行ログ・実データ）を提示できないなら、まだ完了していない。

## いつ使うか

- 「完了しました」「修正しました」「できました」「動くはず」「対応しました」と報告しようとした**直前**
- PR作成・コミット・デプロイの前
- ユーザーに成果物を引き渡すとき
- 「たぶん大丈夫」「問題ないはず」と思ったとき（← 最も危険）

**使わなくてよい**: 雑談・情報提供のみ・調査結果の報告（実装を伴わない）。

## 使い方（30秒の自己照合）

1. 下の**言い訳封じ表**を上から読む
2. 自分の今の状態に当てはまる「言い訳」行があるか確認する
3. 当てはまったら、その行の「とるべき行動」を**実行してから**報告する
4. どの言い訳にも当てはまらない＝証拠が揃っている、なら完了報告してよい

## 言い訳封じ表

失敗カテゴリは CLAUDE.md「自己学習」の4分類（前提ミス / 過剰設計 / 検証不足 / コンテキスト不足）に対応する。

| 言い訳（自分の内心） | 現実 | 失敗カテゴリ | とるべき行動 |
|---|---|---|---|
| 「動くはず」「たぶん大丈夫」 | 検証していない＝ただの推測。これが品質低下の最大原因。 | 検証不足 | 実際に動かす（dev server起動・テスト実行・該当機能を操作）。実挙動を見てから報告。 |
| 「テストは後で書く」 | 後では書かれない。後付けテストは挙動でなく実装を追認するだけ。 | 検証不足 | 変更前に失敗するテスト（再現テスト）を先に書く。Prove-Itパターン（systematic-debugging Phase 4）。 |
| 「コードを読めば正しいのは明らか」 | 読みは仮説。実行結果だけが証拠。 | 検証不足 | 1回でいいので実行して出力を確認。 |
| 「UIはCSS直しただけだから見なくていい」 | レイアウト崩れ・文字重なり・レスポンシブ崩れは目視でしか分からない。 | 検証不足 | dev serverを起動しブラウザ/スクショで確認（rules/frontend.md 動作検証必須）。 |
| 「外部APIは叩けば通るはず」 | レート制限・認証・レスポンス形式は実際に叩くまで不明。本番に書く前に壊れる。 | 前提ミス | サンドボックス/開発用リソースに対して実行し、レスポンスを確認（rules/backend.md）。 |
| 「仕様はこういう意味だと思う」 | 曖昧な要件を黙って埋めた＝認識ズレの温床。 | 前提ミス | 前提を明示し、2件以上の確認事項があればプランモードで確認（feedback_plan_mode_for_questions）。 |
| 「ついでにここも直しておいた」 | 依頼外の変更はレビュー不能・破壊リスク。スコープ逸脱。 | 過剰設計 | 触るのは依頼範囲だけ。別の改善は別タスクとして提案。 |
| 「もっと汎用的に作っておいた」 | 使われない抽象化は負債。boring solution が最善。 | 過剰設計 | 必要最小限のコードに戻す。今必要な分だけ。 |
| 「このプロジェクト特有の事情は知らないが進めた」 | プロジェクト固有の制約（GAS制限・本番ID・トリガー等）を見落とすと事故。 | コンテキスト不足 | 該当プロジェクトのCLAUDE.md「Known Pitfalls」とmemory `project_*.md` を確認。 |
| 「エラーは握りつぶした、たぶん起きない」 | サイレント失敗は本番で人を困らせる。 | 検証不足 | 構造化ログ＋UI通知を入れる（rules/backend.md エラーハンドリング）。 |
| 「ビルドは通ると思う」 | 型エラー・lintは実行しないと分からない。 | 検証不足 | `tsc --noEmit` / lint / build を実行。デプロイ前は /ship-check。 |
| 「全部直した」（テスト未実行） | 実行していない「全部」は願望。 | 検証不足 | 全テストスイートを実行し、PASS出力を提示。 |

## Red Flags（この兆候が出たら止まる）

以下を考えた/やろうとしたら、報告をやめて上表の「とるべき行動」へ:

- テストやビルドを一度も走らせずに「完了」と書こうとしている
- UIを変えたのに画面を一度も見ていない
- 外部API書き込み（POST/PUT/PATCH/DELETE）を本番リソースで未検証のまま通そうとしている
- 「時間がないから検証は省く」と思っている（← 検証省略は手戻りで結局遅くなる）
- ユーザーから「動かないけど？」と言われた直後に、また検証せず別の修正を出そうとしている
- 「スクショ撮るほどでもない」と検証の証拠提示を省こうとしている

## 証拠要求チェックリスト（タスク種別ごと）

報告に**この証拠を添えて**初めて完了とする。

- **コード変更全般**: テスト出力（PASS）／該当機能の実行結果
- **フロントエンド/UI**: dev server起動 → ブラウザ動作確認 or スクリーンショット（rules/frontend.md）。文言・単一スタイル値のみの変更は差分提示＋確認ログで可
- **バックエンド/外部API**: サンドボックスでの実行ログ／レスポンス確認。秘密情報はログに出さない（rules/backend.md）
- **GAS/Slack Bot**: clasp push後の動作確認手順 or トリガー影響確認（rules/backend.md）
- **デプロイ前**: `/ship-check`（テスト/lint/型/ビルド/秘密情報/破壊的変更）を実行しPASS一覧を提示
