### So What? / Why So?（ソーホワット・ワイソー）★

- **一言**: 結論と根拠を双方向に検証し、「言いたいことは何か」と「なぜそう言えるのか」を同時に固める論理確認技法。
- **使う場面（トリガー）**:
  - 資料を書いた後「で、結局どういうこと？」と自分でも言えない
  - 説明後に「根拠は？」「だから何？」と返された経験がある
  - 事実・データは並べたが主張への橋渡しが曖昧
  - ピラミッドを組んだ後、縦の論理を点検したい
- **手順**:
  1. 主張（結論）を1文で書く
  2. 「Why So?（なぜそう言える？）」で直下の根拠・事実を列挙
  3. 根拠群に「So What?（まとめると何が言える？）」と問い、主張が再現されるか確認
  4. 主張⇔根拠が矛盾なく往復できれば論理が通っている
  5. 通らなければ根拠を追加・差替え、または主張を修正
- **なぜ効率的か**: 演繹方向（上→下）と帰納方向（下→上）を1セットで点検でき、「根拠は並ぶが主張が浮く」典型ズレを早期検出。完成後に指摘されるより修正コストが圧倒的に低い。
- **落とし穴**: 根拠の数を増やせば補強できると誤解し矛盾根拠を並べる／So What?を飛ばして箇条書きで終え主張が暗黙のまま／途中階層で往復をやめ全体が未検証。
- **相性**: [[ピラミッドストラクチャー]]（縦の論理検証）、帰納法・演繹法（両方向の構造選択）、空・雨・傘（各ステップでSo What?を問う）

### 帰納法（Inductive Reasoning）★

- **一言**: 複数の個別事実から共通パターンを抽出し一般的な主張を導く（データが先、結論が後）。
- **使う場面（トリガー）**: 複数のインタビュー・数値・事例から傾向を言語化したい／先行事例のない領域で自前データから仮説を立てたい／直感を根拠ある主張に昇格させたい。
- **手順**:
  1. 個別事実・データを列挙（最低3件、できれば5件以上）
  2. 共通要素・パターンを抽出
  3. 「共通して言えることは？」で一般命題を生成
  4. 例外・反証がないか確認し適用範囲を限定
  5. 「確からしい」推論であって「確実」ではないと明記
- **なぜ効率的か**: 定説のない領域でデータから素早く仮説生成できる。演繹が使えない状況でも前進できる。
- **落とし穴**: n=2〜3で一般化する「早まった一般化」が最多の誤用／都合の良い事例だけ集める確証バイアス／反証1件で崩れることを忘れる。
- **相性**: So What?（事実群→主張の動きそのもの）、なぜなぜ分析（帰納で仮説→なぜなぜで因果を掘る）

### 演繹法（Deductive Reasoning）★

- **一言**: 一般的前提（大前提）＋個別事実（小前提）から必然的に結論を導く。確実性は前提の正しさに依存。
- **使う場面（トリガー）**: 業界の定説・社内ルール・確立した法則が手元にある／反論されにくい前提から正当性を示したい／「なぜこの行動を取るべきか」を納得させたい。
- **手順**:
  1. 大前提（真とされる命題）を明示
  2. 小前提（今回の個別状況）を明示
  3. 「大前提×小前提→結論」の三段論法で導く
  4. 大前提が本当に正しいか（出典・根拠）を確認
  5. 論理の飛躍がないか点検
- **なぜ効率的か**: 前提さえ合意できれば結論への道筋が必然になり説明コストが低い。個別データを揃えずに主張できる場面がある。
- **落とし穴**: 大前提が崩れている（古い・例外多）のに未検証で使う→結論が誤り／前提を多段連結し途中の1つの崩壊に気づかない／事例の羅列を「演繹」と誤称。
- **相性**: Why So?（「なぜそう言える?」への回答が演繹構造）、空・雨・傘（一般知識から「雨」を予測する動き）

### 空・雨・傘（Sky-Rain-Umbrella）★

- **一言**: 事実（空）→解釈（雨）→行動（傘）の3段階を分離し、思い込みによる早期結論を防ぐ型。マッキンゼー由来。
- **使う場面（トリガー）**: 事実・解釈・意見が混在し誰が何を根拠に主張しているか不明／「数字が悪い→対策を打て」と反射的議論になっている／報告書が「データ→いきなりアクション」構成／「なんでその結論?」と問われやすい。
- **手順**:
  1. **空（事実）**: 観察できるデータ・出来事を解釈抜きで記述（例「CTRが先月比20%低下」）
  2. **雨（解釈）**: その事実の意味（例「訴求が刺さらなくなっている可能性」）
  3. **傘（行動）**: 解釈に基づく打ち手（例「クリエイティブ刷新＋A/Bテスト」）
  4. 3段階を別の行/スライドに分離し混在を点検
  5. 「傘」が「雨」から論理的に導けるか確認
- **なぜ効率的か**: 事実と解釈の混同という最頻の思考エラーを型で防ぐ。3層分離で議論のどのレベルで割れているか即可視化。
- **落とし穴**: 「空」に解釈が紛れる／「傘」が「雨」から飛躍／順番を並べただけで各段階の精度を確認しない。
- **相性**: So What?（雨→傘の点検）、As is/To be（空=As is、傘実行後=To be）、帰納法（複数の空→共通の雨）

### As is / To be（現状 / 理想像）★

- **一言**: 現状（As is）と理想（To be）を明示しギャップを埋める施策を設計。起点を「解決策の思いつき」でなく「ゴールとの差分」に置く。
- **使う場面（トリガー）**: 改善案検討前に「今どんな状態か」「何のため変えるか」が未言語化／チームのゴール認識がバラバラ／複数問題が混在し優先順位がつかない。
- **手順**:
  1. To be（理想）を先に、測定可能な形で定義（「売上増」でなく「3ヶ月後CVR1.5倍」）
  2. As is（現状）をデータと共に記述
  3. 差分（ギャップ）を特定し原因を分類
  4. ギャップを埋めるアクションを設計し優先順位付け
  5. 実行後に再度As isを測定し接近度を確認
- **なぜ効率的か**: To beを先に固定するとアクションの合否判定基準が明確になり、手段の目的化を防ぐ。
- **落とし穴**: To beを曖昧に定義（測定不能だとギャップ分析に使えない）／As is調査が甘く思い込みで現状記述／How検討に時間をかけすぎTo be定義に戻れない。
- **相性**: 空・雨・傘（As is=空の規律）、なぜなぜ分析（ギャップの根本原因）、コントロール可能/不可能（原因を変えられるかで仕分け）

### なぜなぜ分析（5 Whys）★

- **一言**: 表面的原因で止まらず「なぜ?」を連鎖させ真因を特定。トヨタ生産方式・大野耐一が実践。
- **使う場面（トリガー）**: 同じ問題が繰り返す（対症療法の連続）／「誰かのミス」で調査が終わり再発防止が浮かばない／打ち手が表面的で効いていない／複数仮説から最も根本的な原因を絞りたい。
- **手順**:
  1. 問題を具体的・事実ベースで定義（「成果が出ない」でなく「先月CVRが目標比30%低」）
  2. 1回目の「なぜ?」を推測でなく確認できる事実を根拠に答える
  3. 答えにさらに「なぜ?」を、因果が説明できなくなるまで繰り返す（目安5回・現実は3〜7回）
  4. 各「なぜ→答え」を逆向き（答え→なぜ）で因果成立を確認
  5. 真因到達後、仕組み・プロセス側の再発防止策を設計
- **なぜ効率的か**: 対症療法と根本対処を峻別。「同じ問題の再発」という時間コストを構造改善で根絶でき投資対効果が高い。
- **落とし穴**: 個人の意識・注意力で連鎖を止める（「気をつける」で終わり実効性ゼロ／仕組み側へ掘る）／結論ありきで証拠集め／5回を目標化し論理が飛躍。
- **相性**: As is/To be（ギャップの根本原因）、空・雨・傘（空の精度が出発点の質を決める）、コントロール可能/不可能（真因が自社で変えられるか判断）、ロジックツリー（Whyツリーで分岐を並列管理）

### 5W1H

- 使う場面: 伝達・指示・報告に抜けや曖昧さがあるとき。誰が読んでも同じ行動を取れる状態にしたいとき。
- 手順: When/Where/Who/What/Why/How を埋め、「この情報だけで実行できるか」を第三者目線で確認。

### 6W2H

- 使う場面: 5W1Hで「対象顧客(Whom)」「予算(How much)」が抜けがちなマーケ施策・事業企画。実施主体(Who)と受益者(Whom)を分離したいとき。
- 手順: 5W1H＋Whom(誰に)＋How much(いくらで)の8要素。「Who≠Whom」を意識して別欄で定義。

### コントロール可能 / コントロール不可能

- 使う場面: 問題が複数あり取り組む優先順位とスコープを決めたいとき。「どうにもならないことに消耗している」シグナルが出たとき。
- 手順: 論点を全書き出し→各項目を「自分・自社の行動で変えられるか」で二択分類→可能なものに絞り解決策設計、不可能なものは「与件として受け入れどう適応するか」に切替。
