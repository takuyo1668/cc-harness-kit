# raw/02 整理・構造化と優先順位

思考プロセスマップの「② 整理・構造化する／優先順位を決める」。
★=コア（早見表で即使える／ここは深掘り版）。

### MECE（Mutually Exclusive, Collectively Exhaustive）★

- **一言**: 対象を「漏れなく・ダブりなく」分割し、思考の抜け穴と冗長性を同時に除去する原則。
- **使う場面（トリガー）**: 「何から手をつけるか分からない」／施策を列挙したが「他にもあるはず」の不安が残る／ロジックツリー・競合分析・セグメントを作る前の土台設計／会議でバラバラに出た論点を整理したい。
- **手順**:
  1. 分類対象を1文で確定（例「離脱率が高い原因」）
  2. 分類の軸（切り口）を1つ決める（例「フェーズ別: 流入・閲覧・カート・決済」）
  3. 各分類が重複しないか（ME）を確認
  4. 足し合わせて全体をカバーするか（CE）を確認
  5. 漏れたら切り口を変えるか「その他」を明示
  6. 確定後、各要素に事実・データを当てる
- **なぜ効率的か**: 漏れによる誤判断と、ダブりによる議論の繰り返しを同時に排除。枠組みを先に合意するためチームの認識ズレが起きにくい。
- **落とし穴**: 切り口の選択ミス（本質的差異が出ない軸）／完璧主義の思考停止（「その他」で前進が正解）／MECEと正しさの混同（分類できても中身が事実でなければ誤る）／プレゼンへの過剰適用（重要メッセージが埋もれる）。
- **相性**: [[ロジックツリー]]（MECE維持で階層展開）、[[ピラミッドストラクチャー]]（根拠のダブり点検）、意思決定マトリクス（評価基準の網羅性チェック）

---

### ロジックツリー（Logic Tree）★

- **一言**: 問いを頂点に、MECEを保ちつつ原因・要素・解決策を枝状に展開し、全体構造と打ち手候補を同時に可視化する。
- **使う場面（トリガー）**: 「原因が複数ありそうでどれが本当か判断できない」／施策アイデアが偏っている気がする／KPI未達でどこから改善か不明／提案書の論理構造を整理したい。
- **手順**:
  1. ツリーのタイプを選ぶ（下表。「どんな問いに答えたいか」で決める）
  2. 頂点に問い・テーマ・KGIを1文で置く
  3. 第2層に頂点をMECEに分解した要素を並べる
  4. 各要素を「具体的行動に落とせるか?」を問いながら分解
  5. 施策・仮説・アクションに到達したら止める
  6. 全枝を通覧し漏れ・重複を点検
- **ツリーの4種類**:

  | 種類 | 問い | 用途 |
  |---|---|---|
  | Whatツリー（要素分解） | 何で構成されるか | 全体構造把握・概念整理 |
  | Whyツリー（原因追求） | なぜそうなるか | 真因特定。5Whyと併用で深掘り |
  | Howツリー（問題解決） | どうすれば達成できるか | 施策の網羅的洗い出し |
  | KPIツリー | KGI分解でどの指標が先行か | 数値目標をドライバー指標に展開 |

- **なぜ効率的か**: 散在情報を構造として外部化し全体を見渡せる。どの枝のどの深さに問題があるか特定でき、調査・施策の投資先が絞れる。
- **落とし穴**: 同一階層に異なる切り口が混在（1階層1切り口が原則）／深掘りしすぎ（3〜4階層で十分、仮説が出た枝だけ深く）／Whyを使うべき所でWhatを使う／作ることが目的化（検証しない地図は無価値）。
- **相性**: [[MECE]]（分岐の漏れダブり）、[[ピラミッドストラクチャー]]（Howツリー→報告書）、緊急度/重要度マトリクス（施策候補の絞り込み）

### ピラミッドストラクチャー（Pyramid Principle）★

- **一言**: 「結論→根拠→事実」の3層で組み、相手が聞きたい順に伝えメッセージ通過率を最大化する。バーバラ・ミント体系化。
- **使う場面（トリガー）**: 報告書・提案書・スライドの論理構成を固める／「何が言いたいか分からない」と言われた／根拠データは集まったが並べ方に迷う／ロジックツリーで施策が出た後の説明資料化。
- **手順**:
  1. 結論（キーメッセージ）を1文で、聞き手が最も知りたい答えを先に置く
  2. 「なぜそう言える?」で根拠を2〜4個列挙（根拠同士がMECEか確認）
  3. 各根拠に事実・データを1〜2個ずつ紐付け（実数値・出典明記）
  4. 縦の論理（So What/Why So）で飛躍を点検
  5. 横の論理（同階層のダブり・並列性）を点検
- **なぜ効率的か**: 結論先行で聞き手にコンテキストができ細部も定着。作り手は根拠の抜けを構造で発見できる。
- **落とし穴**: 結論ありきで事実を都合よく選ぶ（反証確認必須）／Rule of 3への固執（数より質、2〜4個）／結論未確定なのにトップダウン強行（先にWhyツリーで分析）／1スライドに複数キーメッセージ。
- **相性**: [[MECE]]（根拠の横並び点検）、[[ロジックツリー]]（Howツリー→提案書）、意思決定マトリクス（選定結果→説明）

### 緊急度/重要度マトリクス（アイゼンハワーマトリクス）★

- **一言**: タスクを重要度×緊急度の4象限に分け、今すぐやるべき仕事と捨てる仕事を即判断する。
- **使う場面（トリガー）**: タスクが積み上がり何からか判断できない／忙しいのに成果が出ない／週次棚卸でやめるものを決めたい／チームで優先順位の認識合わせ。
- **手順**:
  1. タスクを全書き出し（頭の外に出す）
  2. 縦=重要度（ゴール/KGIへの影響）、横=緊急度（締切の近さ）
  3. 4象限に分類:
     - 第1（重要×緊急）: 即実行
     - 第2（重要×非緊急）: スケジュール化して計画実行（最重要ゾーン）
     - 第3（非重要×緊急）: 委任or短時間処理
     - 第4（非重要×非緊急）: 排除
  4. 第2象限に実行日時を確保
  5. 週次で再分類
- **なぜ効率的か**: 「緊急＝重要」という脳の自動判断を打ち破れる。第2象限に時間を使えると中長期成果が改善。「何をやめるか」を能動決定でき総量が減る。
- **落とし穴**: 重要度の軸が外圧でずれる（KGI貢献度で判断）／全部が第1象限に集まる（基準が甘いサイン）／第3象限を自分でやり続け第2象限が潰れる／粒度が不揃い。
- **相性**: [[ロジックツリー]]（Howツリーの施策を絞る）、意思決定マトリクス（重み付けで順位決定）、MECE（棚卸の分類漏れ）

### 意思決定マトリクス（Weighted Decision Matrix）★

- **一言**: 複数選択肢を、重みの異なる複数基準でスコアリングし合計で比較、感覚でなく論拠で最善を選ぶ。
- **使う場面（トリガー）**: 「どのツール/施策を選ぶか」で各軸にトレードオフがある／直感で決めた後に理由を説明できない／判断軸がバラバラで意見が収束しない／投資・採用・技術選定など後検証が必要。
- **手順**:
  1. 選択肢を横軸に並べる
  2. 評価基準を縦軸に列挙
  3. 各基準に重みを設定（合計100に配分）
  4. 各選択肢×各基準を1〜5でスコア
  5. 重み×スコア=加重スコアを合計
  6. 最高スコアを候補に、「この結果に違和感はないか」を最終確認（あれば重みを見直す）
- **なぜ効率的か**: 数値化で「なんとなく良い」の言語化を強制。重みの合意プロセスで「何を最優先か」の本質議論が先に行われる。根拠が表で残り後検証が容易。
- **落とし穴**: 選択肢ありきで基準を後付け（基準・重みは選択肢を見る前に）／重み合計が100にならない・均等配分（優先順位なしと同じ）／定性情報の無理な数値化／算出後の違和感チェック省略。
- **相性**: 緊急度/重要度マトリクス（第2象限の施策を精緻化）、[[MECE]]（基準の網羅性）、[[ピラミッドストラクチャー]]（選定結果→提案）

### 課題設定シート

- 使う場面: 「問題は分かるが何から解くべきか」が曖昧、または作業前に解くべき問い（イシュー）を固めたいとき。問題（現状と理想の差）と課題（差を埋める取り組み対象）の混同防止。
- 手順: (1)テーマ（問題）を1文、(2)最も解くべき課題を1つに絞る、(3)6W2Hで整理し担当・期限・成功基準まで明記。1シート1課題で運用。

### マトリクス図（2×2マトリクス）

- 使う場面: 比較対象を2軸で俯瞰しポジショニングや分布パターンを発見したいとき。軸の選択が成果の全て。
- 手順: (1)答えたい問いを確定、(2)対応する2軸を選ぶ（例 インパクト×実現難易度）、(3)対象を配置し集積/空白象限からインサイトを読む、(4)問いに答えない軸なら変えて再作成。BCGマトリクスもこの特定用途版。

### ベン図（Venn Diagram）

- 使う場面: 複数集合の重なり・差分・唯一性を視覚化したいとき（セグメント定義、差別化領域、要件整理）。MECEが「分離」なのに対しベン図は「重なりを意図的に見せる」用途。
- 手順: (1)円の数=集合数（2〜3が実用的）、(2)ラベル付け、(3)重なり領域に両属性を記述、(4)排他領域で差分を明確化。
