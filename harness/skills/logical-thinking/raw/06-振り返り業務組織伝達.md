# raw/06 振り返り・業務可視化・改善・組織・伝達

思考プロセスマップの「⑦ 振り返る・改善する／⑧ 伝える」＋業務・組織・人。
★=コア（早見表で即使える／ここは深掘り版）。

---

### PDCA（Plan / Do / Check / Act）★

- **一言**: 「計画→実行→評価→改善」を循環させ継続的に成果を高めるマネジメントサイクル。
- **使う場面（トリガー）**: 施策結果が目標を下回った／新規プロジェクトを定期モニタリングしたい／月次・週次レビューの構造化。
- **手順**:
  1. Plan: 数値目標と達成手段を具体化（KPI・期限・担当を明記）
  2. Do: 計画どおり実行し実績データを記録
  3. Check: 実績と目標を比較し差異の原因を特定（なぜなぜで深掘り）
  4. Act: 原因への改善策を決定し次のPlanに反映
- **なぜ効率的か**: 評価と改善を定型サイクルに組み込み、個人の経験値でなく組織として学習できる。
- **落とし穴**: CheckとActを混同し「改善策を出して終わり」（Checkは差異分析のみ、Actは後工程）／Doのデータを取らずCheckが主観に。
- **相性**: [[KPT]]（振り返り粒度を下げる）、ECRS（Actの改善策立案）、SMART（Plan精度向上）

### KPT（Keep / Problem / Try）★

- **一言**: 「続ける・やめる・試す」の3分類で振り返りを構造化し次のアクションを導く。
- **使う場面（トリガー）**: スプリント末・プロジェクト終了後・月次レビュー。特に「何が良くて何が悪かったか言語化できていない」とき。
- **手順**:
  1. Keep: 継続すべき成功行動・良い習慣を列挙
  2. Problem: 障害・改善すべき事象を列挙（原因でなく事象を先に）
  3. Try: Keepを伸ばす施策、またはProblemを解消する具体改善案
  4. 次サイクル開始時にTryの実施状況を確認し次のKeep/Problemへ昇格
- **なぜ効率的か**: 批判で終わらず「次に試すこと」を必ず出す構造で、ポジティブな行動変化につながる。
- **落とし穴**: Tryに「もっと頑張る」など行動不能な抽象案が混入（「誰が・いつまでに・何を」に落とす）／Problemを人格批判に使う。
- **相性**: [[PDCA]]（TryをPlanに変換）、YWT（個人の内省に使い分け）

### PREP（Point / Reason / Example / Point）★

- **一言**: 結論を先に述べ理由・具体例で補強し再び結論で締める論理的伝達テンプレート。
- **使う場面（トリガー）**: 提案・報告・メール・会議発言。「何が言いたいか分からない」と言われる構造の矯正。
- **手順**:
  1. Point（結論）: 最も伝えたいことを1文で先に
  2. Reason（理由）: なぜその結論か、根拠2〜3点
  3. Example（具体例）: データ・事例で理由を補強
  4. Point（再提示）: 「以上の理由から〇〇です」で締める
- **なぜ効率的か**: 聞き手が最初に結論を受け取り、以降を「裏付けか反論か」の文脈で処理でき、認知負荷が下がり合意が速い。
- **落とし穴**: ExampleとReasonを混同し「事例だけ3つ」になる／Point自体が曖昧だと全体が崩れる。
- **相性**: [[ピラミッドストラクチャー]]（同じ結論先行構造）、TAPS（提案文のマクロ構成）

### YWT（やったこと / わかったこと / つぎにやること）

- 使う場面: 個人・チームの日次〜週次の振り返り（KPTより軽量）。
- 手順: Y=実行した事実を列挙→W=得た学びを言語化→T=Wから次の行動を決定。「事実→内省→行動」固定で振り返りが浅くなりにくい。

### 業務棚卸シート

- 使う場面: 組織変更・役割再設計・業務改善の前処理として現状業務を全量可視化。
- 構成: 業務名・頻度・所要時間・担当者・関連システム・ムダ度合いの列で一覧化。ECRS/RACI/業務フロー図の入力データになる。

### 業務フロー図

- 使う場面: 業務手順の引き継ぎ・ボトルネック特定・システム要件定義。
- 構成: 開始〜終了を矩形（作業）・菱形（判断）・矢印で表現。スイムレーンで担当部門を分けると「誰がどこで止めているか」が分かる。

### PERT図

- 使う場面: 複数タスクが並走・依存するプロジェクトのスケジュール管理。
- 構成: タスク=ノード、依存=矢印。クリティカルパス（最長経路）を特定し、どこに余裕がありどこを遅らせてはいけないかを判断。

### RACI

- 使う場面: プロジェクト開始時・組織変更後の役割の曖昧さ解消。
- 構成: タスク×関係者のマトリクスで R=実行責任者 / A=最終説明責任者（1タスク1名） / C=相談される人 / I=情報共有される人。AとRの混同が最多の失敗。

### ムリ・ムラ・ムダ（ダラリの法則）

- 使う場面: 業務改善・生産性向上の現状分析。
- 構成: ムリ（過負荷）・ムラ（ばらつき）・ムダ（価値を生まない余剰）の3軸。ムラを放置するとムリとムダが両方発生するためムラ解消が優先。

### ECRS

- 使う場面: 業務棚卸後の改善優先順位付け。
- 構成: E=Eliminate(排除)→C=Combine(結合)→R=Rearrange(順序変更)→S=Simplify(簡略化)の順。左ほど効果大なので順番どおり問うのが重要。

### 業務改善提案シート

- 使う場面: 改善施策を上司・経営層に承認させる。
- 構成: 現状課題（数値化）・改善案・期待効果（工数/コスト/品質）・必要リソース・スケジュールを1枚に。ECRSやムリムラムダの分析結果を落とし込むと説得力が増す。

### ミッション・ビジョン・バリュー（MVV）

- 使う場面: 組織の方向性共有・採用・意思決定基準の設計。
- 構成: ミッション=存在意義／ビジョン=目指す将来像／バリュー=行動規範の3層。ミッションが根幹、バリューは日常判断基準に落とす。

### Will/Can/Must

- 使う場面: 個人のキャリア設計・業務アサイン・面談での目標設定。
- 構成: Will（やりたい）×Can（できる）×Must（求められる）の3円。重なりが「成果と動機が両立する領域」。薄い場合に何を変えるか議論。

### Need/Wantマトリクス

- 使う場面: タスク優先順位・業務アサイン最適化。
- 構成: 縦=Want（個人の意欲）×横=Need（組織の必要性）の4象限。高Need×高Want（優先）／高Need×低Want（動機向上）／低Need×高Want（後回し）／低Need×低Want（廃止候補）。

### ジョハリの窓

- 使う場面: チームビルディング・1on1・フィードバック文化の醸成。
- 構成: 自分×他者の「知っている/知らない」の4象限（開放・盲点・秘密・未知）。自己開示とフィードバックで「盲点の窓」を縮小し相互理解を深める。

### 認知/行動ループ（ABCモデル）

- 使う場面: 思考パターンの改善・コーチング・メンタルヘルス対策。
- 構成: A（出来事）→B（認知・信念）→C（感情・行動結果）。同じ出来事でもBが変わると結果が変わる。認知の歪みを特定し書き換えるのが介入点。

### ウォント/コミットメント

- 使う場面: チーム結成・プロジェクト開始時の役割分担と相互理解。
- 構成: ウォント=他者への期待／コミットメント=自分が提供できる貢献。全員が書き出し共有し、期待のすれ違いと強みの重複・空白を可視化。

### PM理論

- 使う場面: リーダーのスタイル診断・マネジャー育成。
- 構成: P機能（目標達成: 指示・督促）とM機能（集団維持: 配慮・承認）の強弱で4タイプ（PM型=理想／Pm型=成果偏重／pM型=関係偏重／pm型=両低）。三隅二不二, 1966。

### ステークホルダー分析

- 使う場面: プロジェクト推進・組織変更・新施策導入前に関係者の影響力と利害を整理。
- 構成: 縦=影響力×横=関心度の4象限。高影響×高関心（徹底管理・密に）／高影響×低関心（関心を高める）／低影響×高関心（情報提供）／低影響×低関心（最小限）。

### マズローの欲求五段階説

- 使う場面: 採用・定着・動機付け施策の設計、エンゲージメント分析。
- 構成: 生理的→安全→社会的（帰属）→承認→自己実現の5段階。下位が満たされると上位が動機に。職場では「給与→雇用安定→帰属感→評価→成長・挑戦」。

### 動機付け・衛生理論（ハーズバーグ二要因理論）

- 使う場面: 満足度調査の設計・離職要因分析・評価/福利厚生制度の設計。
- 構成: 衛生要因（給与・労働条件・環境＝不満を取り除く）と動機付け要因（達成・承認・成長＝満足を生む）を分離。衛生要因を整えても意欲は上がらない（ゼロに戻るだけ）が最重要の示唆。

### Will/Skillマトリクス

- 使う場面: メンバーへの指導方針・アサイン・育成計画。
- 構成: 縦=Will（やる気）×横=Skill（能力）の4象限。高Will×高Skill（任せる）／高Will×低Skill（育成・OJT）／低Will×高Skill（動機向上）／低Will×低Skill（関係構築が先）。

### GROWモデル

- 使う場面: 1on1・コーチング面談・課題解決支援。
- 手順: G=Goal（達成したいこと）→R=Reality（現状）→O=Options（選択肢）→W=Will（意志・行動宣言）。コーチが答えを与えず質問で本人に導かせる構造。

### 商品企画書

- 使う場面: 新商品・新機能の上申・社内承認取得。
- 構成: 背景（市場・課題）・コンセプト・ターゲット・提供価値・仕様・想定価格・販売計画・収支見通し・スケジュール。3Cの根拠を入れると説得力が増す。

### イベント企画書

- 使う場面: 社内外イベントの企画提案・予算申請。
- 構成: 目的・ゴール（KPI）・ターゲット・日時・場所・プログラム・必要リソース・リスクと対策・期待効果。ゴールを「参加後の行動変容」まで定義できると承認率が上がる。

### TAPS法（To be / As is / Problem / Solution）

- 使う場面: 問題提起型のプレゼン・提案資料の構成設計。
- 構成: T=To be（理想）→A=As is（現状）→P=Problem（ギャップ＝問題）→S=Solution（解決策）。理想と現状のギャップで問題の必然性を感じさせてから解決策を出すため押し付けにならない。※David RockのTAPS（Tell/Ask/Problem/Solution）はコーチング領域の別FW。文脈で定義を確認。

### PASONAの法則

- 使う場面: セールスコピー・LP・営業トークの構成設計。
- 構成: P=Problem（問題提起）→A=Affinity（親近感・共感）→S=Solution（解決策）→O=Offer（提案・特典）→N=Narrowing down（限定性・緊急性）→A=Action（行動喚起）。顧客心理に沿った順番で読み飛ばされにくい。
