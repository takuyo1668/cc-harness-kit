# 観点別チェックリスト（検証可能条件のみ）

design-reviewの各エージェントが参照する観点別チェック項目。
**全項目を「測定可能・検証可能」な条件で記述する**。「良い」「適切な」などの曖昧語は禁止。

---

## A. UI品質（ui-specialist 担当）

### A-1. 視覚階層

- [ ] h1/h2/h3 のフォントサイズに **1.4倍以上のジャンプ** がある（h1=32px, h2=24px, h3=18px など）
- [ ] 同一画面内のテキストサイズは **5種類以下** に収まっている
- [ ] CTA は周囲のテキスト/ボタンより **コントラスト比 3:1 以上** で目立つ

### A-2. 余白とリズム

- [ ] 余白は **8の倍数** で統一（8/16/24/32/48/64）
- [ ] section間の vertical padding は **48px以上**（モバイルでも32px以上）
- [ ] 隣接要素間の余白は **要素サイズの50%以上** ある（タイトルと本文、ボタンと説明文など）

### A-3. タイポグラフィ

- [ ] フォントファミリーは **2種類以下**（見出し用 / 本文用）
- [ ] 行間 (line-height) は本文 **1.5〜1.7**、見出し **1.2〜1.4** の範囲内
- [ ] フォントweightは **3段階以下**（例: 400 / 600 / 700）

### A-4. 色

- [ ] メインカラー + アクセント + グレースケール の **3系統以内** に収まっている
- [ ] hex直打ちではなく **CSS変数 or Tailwind トークン** で統一されている
- [ ] hover/focus/disabled の色がそれぞれ定義されている

### A-5. コンポーネント一貫性

- [ ] 同種コンポーネント（ボタン/カード/入力欄）の角丸が **統一されている**
- [ ] アイコンサイズが **3種類以下**（16/20/24 など）
- [ ] アイコンとテキストの間の余白が **統一されている**（4px or 8px）

---

## B. 美的整合性（ui-specialist 担当・preferences.md 参照）

このセクションは `preferences.md` の内容を動的に注入する。
固定チェック項目ではなく、ユーザーの確立した好み傾向と一致しているかを評価する。

評価方法:
- preferences.md の「確立した好み」全項目について「一致/不一致/該当なし」を判定
- 一致率を `美的整合性スコア = 一致 / (一致 + 不一致) × 10` で算出

---

## C. UX / 情報設計（product-designer 担当）

### C-1. 情報優先順位

- [ ] ファーストビュー（モバイル: 上から600px以内）に **何のサイトか・何ができるか・次のアクション** の3要素が揃っている
- [ ] CTAは ファーストビュー内に **1つ以上**、ページ全体で **3つ以下** に絞られている
- [ ] 1ページに **1つの主要目的** が明確（複数目的の混在は減点）

### C-2. 認知負荷

- [ ] 1ブロックのテキスト量は **3行以下 or 100文字以下**（モバイル基準）
- [ ] 選択肢提示は **7つ以下**（Hick's Law）
- [ ] フォームの入力項目は **必須のみ**、任意項目は折りたたみ or 別ステップに分離

### C-3. フィードバック

- [ ] ボタンクリック時の **hover/active/disabled 状態** が視覚的に区別できる
- [ ] フォーム送信中の **ローディング表示** がある
- [ ] エラー時の **メッセージが該当フィールドの近く** に出る（離れた場所のサマリだけはNG）

### C-4. エンプティ/エラー状態

- [ ] 空の状態（データなし）に **次のアクション提案** がある
- [ ] 404/500等のエラー時にも **戻る導線** がある
- [ ] フォームバリデーションは **入力中（onBlur）に表示**、submit後だけはNG

### C-5. マイクロコピー

- [ ] CTAの文言が「送信」「ログイン」等の汎用語ではなく **動詞+目的語** になっている（例: 「無料で試す」「LINEで質問する」）
- [ ] エラーメッセージが **原因+解決策** を含む（「エラーが発生しました」のみはNG）
- [ ] 専門用語に **括弧書きの説明 or ツールチップ** がある

## D. アクセシビリティ（tech-researcher 担当・WCAG 2.1 AA 準拠）

### D-1. コントラスト（WCAG 1.4.3）

- [ ] 通常テキスト: 背景とのコントラスト比 **4.5:1 以上**
- [ ] 大きいテキスト（18pt以上 or 14pt bold以上）: **3:1 以上**
- [ ] UIコンポーネント・グラフィカルオブジェクト: **3:1 以上**

### D-2. キーボード操作（WCAG 2.1.1）

- [ ] すべてのインタラクティブ要素が **Tab キーで到達可能**
- [ ] フォーカス時の視覚インジケータが **2px以上のアウトライン or 同等の表示**
- [ ] Tab順序が **視覚的な順序と一致**（DOM順序＝視覚順序）

### D-3. セマンティックHTML（WCAG 1.3.1）

- [ ] 見出しが `h1` → `h2` → `h3` の **階層順** で使われている（飛ばしNG）
- [ ] ボタンが `<button>`、リンクが `<a>` で実装されている（`<div onclick>` はNG）
- [ ] フォーム要素に `<label>` が **明示的に紐付いている**（for属性 or 親要素として）

### D-4. 画像と代替テキスト（WCAG 1.1.1）

- [ ] 意味のある画像に **alt属性** がある
- [ ] 装飾画像は **alt=""** または `role="presentation"` で明示
- [ ] アイコンボタンに **aria-label** で操作内容を記述

### D-5. タップ領域（WCAG 2.5.5）

- [ ] タップ可能領域は **最小 44x44px**（モバイル）
- [ ] 隣接するタップ要素間に **8px以上の余白** がある

### D-6. ARIA（WCAG 4.1.2）

- [ ] モーダル/ダイアログに `role="dialog"` と `aria-modal="true"`
- [ ] アコーディオン等の開閉状態に `aria-expanded`
- [ ] ライブリージョン（通知/エラー）に `aria-live="polite"` or `"assertive"`

---

## E. 動的注入セクション

以下は patterns.md から動的に追記される:

- **採用パターン** が `patterns.md` の `## 採用パターン` から各エージェントへ「重要観点」として注入される
- **不採用パターン** が `## 不採用パターン` から各エージェントへ「指摘しない観点」として注入される

これにより、回を重ねるごとにチェックリストがユーザー好みに最適化されていく。
