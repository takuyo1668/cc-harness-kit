# Design Review — 自己進化する多角デザインレビュー

UI/UXの専門家3名（並列）で実装コードをレビューし、フィードバックを毎回蒸留することで「**使うごとにユーザーの美的好みを学習して精度が上がる**」レビュースキル。

## 基本原則

- **多角評価**: UI品質 / UX / アクセシビリティ / 美的整合性 を独立した専門家視点でレビュー
- **PDCA**: 毎回レビュー後にユーザーFBを取り、`preferences.md` と `patterns.md` に蒸留 → ノイズが減り精度が上がる
- **検証可能条件のみ**: 「良い」「適切」など曖昧語は禁止。「タップ領域44x44px以上」のような測定可能条件で評価
- **ユーザー稼働最小**: 中間確認は出さない。最終1回のFB収集のみ
- **学習結果を必ず注入**: 各専門家エージェント起動時に `preferences.md` を読ませて好みを内部化させる

---

## 引数

- `<path>`: 必須。レビュー対象のファイル/ディレクトリ。HTML/CSS/JSX/TSX。複数指定可
- `--quick`: 任意。3エージェント並列ではなく ui-specialist 1名で簡易レビュー
- `--no-feedback`: 任意。Phase 4（FB収集）と Phase 5（蒸留）をスキップ。学習しない

例:
- `/design-review ~/your-project/app/page.tsx`
- `/design-review ~/your-project/components/ --quick`

## ワークフロー

### Phase 0: 好みプロファイル読み込み

実行前に以下3ファイルを必ず読む:

1. `~/.claude/skills/design-review/preferences.md` — ユーザーの美的好み（蓄積データ）
2. `~/.claude/skills/design-review/patterns.md` — 採用/不採用された指摘の履歴
3. `~/.claude/skills/design-review/checklist.md` — 観点別の検証可能条件一覧

読んだ内容を後続Phaseの全エージェントに**プロンプトとして注入する**（直接ファイルパスを渡すのではなく、関連箇所を抜粋してプロンプトに埋め込む）。

### Phase 1: 対象分析

引数で指定された対象を Read で読み込む:

- ディレクトリ指定 → Glob で `*.{html,css,jsx,tsx,vue,svelte}` を再帰収集
- ファイル指定 → そのまま読む
- 画面構造（主要セクション、CTA配置、フォーム有無、レスポンシブ対応）を1段落で要約

ここで対象が見つからない/読めない場合はユーザーに報告して停止。

### Phase 2: 並列レビュー（3エージェント同時起動）

Agent ツールで以下3エージェントを**1メッセージ内で並列起動**する（`--quick` 時は ui-specialist のみ）。

各エージェントへのプロンプトには共通で以下を含める:

```
【対象コード】
（Phase 1で読んだコード全文）

【ユーザーの美的好みプロファイル】
（preferences.md の関連抜粋）

【過去の不採用パターン】
以下は過去にユーザーが「指摘不要」と判断した観点。今回も指摘しない:
（patterns.md の「不採用」セクション抜粋）

【出力フォーマット】
以下のMarkdownで返す:

## スコア
- UI品質: X/10
- 美的整合性（ユーザー好み一致度）: X/10
- 根拠: 1-2文

## 課題リスト
| 優先度 | 課題 | 該当箇所 | 改善案（検証可能） | 美的好み参照 |
|--------|------|----------|--------------------|--------------|
| P0 | ... | line 23-29 | padding: 16px → 24px | preferences.md L12 |

優先度判定:
- P0: 致命的（壊れて見える/CVR直撃/WCAG違反）
- P1: 中程度（一貫性欠落/認知負荷高）
- P2: 軽微（ポリッシュ）
```

エージェント別の追加指示:

**A. ui-specialist** （UI品質＋美的整合性）
- checklist.md の「UI品質」「美的整合性」セクションのみ使用
- 視覚階層 / 余白 / タイポ / 色 / アイコン / コンポーネント一貫性
- preferences.md の好みプロファイルとの一致度を `美的整合性スコア` として明示

**B. product-designer** （UX / 情報設計）
- checklist.md の「UX」セクションのみ使用
- 情報優先順位 / フロー / 認知負荷 / マイクロコピー / エラーリカバリ / エンプティ状態
- 「ユーザーが迷う/詰まる箇所」を line 番号で具体指摘

**C. tech-researcher** （アクセシビリティ）
- checklist.md の「アクセシビリティ」セクションのみ使用
- WCAG 2.1 AA 基準 / コントラスト比 / フォーカス可視 / ARIA / キーボード操作 / タップ領域
- 違反は WCAG 基準番号と該当行を明示

### Phase 3: 統合 + Markdownレポート出力

3エージェントの結果を統合する:

1. **重複指摘の統合**: 同一行への複数指摘は1行にまとめ「指摘元」列に複数記載
2. **優先度の正規化**: P0/P1/P2で並べ替え
3. **トータルスコア**: 3エージェントのスコア平均
4. **Markdownレポート生成**: `templates/report.md` を読み、プレースホルダーを置換して保存
   - **スキル内（蒸留参照用）**: `~/.claude/skills/design-review/reviews/YYYY-MM-DD-{project}.md`
   - （任意）閲覧用に外部ノートアプリ（Obsidian等）へも保存したい場合は、そのパスを自分の環境に合わせて追記する
5. **チャット上にサマリを直接展開**: スコア・P0課題・主要な改善案をその場で見られるように出力（ユーザーが別ファイルを開かなくても判断できる状態にする）

`{project}` はパス先頭のディレクトリ名から推測（例: `~/your-project/app/page.tsx` → `your-project`）。

### Phase 4: フィードバック収集

`--no-feedback` 指定時はスキップ。

`AskUserQuestion` で**全課題を一括で**3択判定させる:

- 課題が4件以下 → 1問につき1課題で AskUserQuestion（最大4問）
- 課題が5件以上 → P0のみAskUserQuestion対象、P1/P2は「P1/P2は全採用しますか？」の一括問のみ

選択肢:
- **採用**: 修正する価値あり。次回も似た指摘してOK
- **不採用**: 指摘がズレている。次回は指摘しない
- **保留**: 今回は対応しないが、指摘自体は妥当

### Phase 5: 蒸留（preferences.md / patterns.md 更新）

FB結果を以下に書き戻す:

**preferences.md 更新ルール**:
- 「採用」が連続2回以上の好み傾向 → `## 確立した好み` セクションに追記
- 例: padding 24px が連続採用 → 「余白は16pxより24pxを好む」

**patterns.md 更新ルール**:
- 「不採用」になった指摘 → `## 不採用パターン` に追記（次回からエージェントに「指摘しない」と注入）
- 「採用」になった指摘 → `## 採用パターン` に追記（次回も同種を検出させる）
- 同一指摘が累計3回以上「採用」または「不採用」になったら、checklist.md 本体に昇格 or 削除を提案

**注意**: 蒸留時にユーザーへの確認は不要。サイレント更新する（学習過程の透明性は patterns.md を直接読めば追える）。

### Phase 6: 完了レポート

最終出力（チャット上）に以下を含める:

```
✅ デザインレビュー完了

📊 統合スコア: X/10
🎯 検出課題: P0=N件, P1=N件, P2=N件
📄 レポート: <スキル内Markdownパス>
📁 ログ: <（任意）外部ノートアプリのパス>
🧠 学習: preferences.md にN件追記、patterns.md にN件追記

次回 /design-review はこの学習を反映します。
```

CLAUDE.md ルールに従い、メタ認知ジャーナル（判断/根拠/次回の省略）も末尾に追記。

## エラーハンドリング

- **対象ファイルが見つからない**: ユーザーに正しいパスを聞いて停止
- **エージェント並列起動失敗**: 1つでも失敗したら、成功した結果だけで Phase 3 に進む（部分提供）
- **テンプレ読込失敗**: テンプレなしで素のMarkdownを生成して保存
- **preferences.md が壊れている / 巨大化**: 1000行超えたら「整理しますか？」とユーザーに確認

## 設計判断（なぜこの構造か）

- **3エージェント並列**: 観点ごとに独立した思考にするため。1エージェントに4観点持たせると観点が薄まる
- **preferences.md を強制注入**: ユーザーの好みは「読めるけど読まれない」状態を防ぐ。プロンプトに直接埋める
- **スキル内にMarkdown保存**: reviews/ は PDCA参照元（grep可能・蒸留時に読みやすい）。外部ノートアプリへの閲覧用コピーは任意
- **不採用パターンを次回に注入**: 指摘ノイズを毎回減らす。これが「使うごとに精度が上がる」の核心
