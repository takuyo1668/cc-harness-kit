---

## このスキルの設計思想（成長型）

このスキルは「使うほど精度が上がる」設計です。
- セッション終了時に「使用後の学習ログ」セクションを更新する
- 次回使う前に過去ログを参照し、品質基準・差し替え観点を反映する
- 関連メモリ: `feedback_market_research_v1.md` を必ず最初に読む

---

## 標準ワークフロー（7フェーズ）

### Phase 1: 仮説立案（PM）
依頼を読み、5本前後の初期仮説を立てる。各仮説は「市場 / センターピン / 局地戦 / 差別化 / マネタイズ」など独立した観点をカバーする。各仮説に「これが真なら事業はこう動く」を1文で添える。

### Phase 2-A: 並列リサーチ（5名同時起動）

依頼の性質に応じて5〜6名を選定し、Agentツールで同時に並列起動する。**run_in_background=true で全員バックグラウンド実行**。

| 役割 | エージェント | 主な担当 |
|---|---|---|
| 市場 | market-researcher | 市場規模・成長性・経済規模・GMV・カテゴリ別単価 |
| ユーザー | ux-researcher | JTBD・痛点・ペルソナ・コミュニティ文化・一次ソース |
| 技術 | tech-researcher | 競合機能ギャップ・MVP工数・防壁・リーガル |
| 獲得 | ad-specialist | チャネル・CAC/LTV試算・ローンチGTM |
| グロース | growth-hacker | コールドスタート・ランチェスター戦略・KPIツリー |
| **競合視点（事業レビュー時は必須）** | general-purpose（"競合プレイヤーロール"指定） | 「自分が競合の経営者ならどうこの新規参入を潰すか」「対抗策3つ」 |

各エージェントへの指示テンプレートは Phase 2-A の最後に記載。

### Phase 2-B: ファクトチェック（PM自身）
- 全員の主要数値をクロスバリデーション（複数ソースで一致するか）
- 推定値・一次ソース・類推の信頼性ラベル
- 数値ソースの古さチェック（3年以上前のデータは「要再検証」と明示）
- 循環論法の検出（複数手法が共通の数値を起点にしていないか）

### Phase 3: 多角レビュー（並列2名）

| レビュー観点 | エージェント |
|---|---|
| UX・ユーザーストーリー成立性 | product-designer |
| 論理欠陥・MECE違反・データ批判 | code-reviewer（戦略コンサル評論家ロール） |

各レビュアーに「総合判定A/B/C」「致命的欠落TOP5」「論理弱点TOP3」「修正提案」を出させる。

**A/B/C判定の運用**:
- A=採用 / B=PM裁量で統合修正してPhase 4へ / C=Phase 2に差し戻し
- 両者A→直接Phase 4
- 片方以上B→PMが指摘12項目程度を統合してv2作成→Phase 4
- C混入→該当領域のリサーチを再起動

### Phase 4: 統合・ビジュアル化（並列2-3名）

- **doc-writer**: Markdownレポート（自分用思考整理は8,000-10,000字）
- **ui-specialist**: HTMLスライド（10-15枚、テイストはユーザー選択）
- **copywriter**: 文章再構成（任意）

**重要**: Writeツールの権限拒否が発生する場合があるため、PM自身が直接書き起こす運用を併用する。Agent側には「設計・構成・本文素案」までを担当させる。

### Phase 5: PM最終レビュー
5観点でA判定を出す:
1. 論理整合性
2. データ信頼性
3. MECE性
4. 実行可能性
5. デザイン品質

### Phase 6: 納品
- 成果物パスを明示
- ハイライト3行（センターピン or 核心結論）
- 次の意思決定ポイント（ユーザーが何を決めるべきか）

### Phase 7: 使用後の学習ログ更新（成長型の核心）
- うまくいった点・改善点を `feedback_market_research_v1.md` に追記
- 新しいリサーチ観点があれば次回テンプレに反映

## デザインテイストのヒアリング

Phase 4でHTMLスライド・レポートを作る前に、デザインテイストをユーザーに4択で確認する。

```
【デザインテイストの提案】（プランモードでヒアリング）
  A. コンサル風（McKinsey/BCG調）— Navy×Accent赤・1スライド1メッセージ
  B. モダンミニマル（Apple/Stripe調）— 白基調・余白多い・タイポ重視
  C. ジャーナリスティック（NYT/Bloomberg調）— Serif混在・引用重視・データビズ強め
  D. ビジュアル重視（インフォグラフィック型）— 図解・アイコン・色彩豊富

参照先: `~/.claude/references/slide-design-repertoire.md`
```

## 各エージェントへの指示テンプレート（Phase 2-A）

各エージェントには以下を必ず含めて指示する:
- 検証すべき仮説（PM Phase 1のうち該当するもの）
- 調査項目（MECEに5-7セクション）
- アウトプット形式（エグゼクティブサマリー・本文・比較表・引用URLリスト）
- 信頼性ラベル（一次/推定/類推の3段階）
- 不確実性が高い数字は3つ以上のソースで三角測量
- 「多い/少ない」等の曖昧表現禁止、具体的な数値・期間・ソース

**Acknowledgment指示も必ず含める**:
```
作業開始前に以下を返せ:
【理解確認】
- 依頼内容: （1文要約）
- アウトプット: （何を返すか）
- 不明点: （あれば。なければ「なし」）
```

## 競合視点（devil's advocate）の運用

事業レビュー・新規事業の戦略立案時は **「もし自分が競合の経営者ならどう潰すか」** を必ず1名のエージェントに担当させる。指示テンプレ:

```
あなたは [Skeb/BOOTH/VRChat公式] の経営者として、本事業（[依頼の事業]）が市場参入してきた場合の対抗策を3つ設計せよ。

- 短期(0-3ヶ月)：価格・機能・パートナー戦略のどれで対抗するか
- 中期(3-12ヶ月)：本気で潰しに行く場合の打ち手
- 長期(12ヶ月以降)：吸収合併・市場棚上げ・規制ロビー等の選択肢

各打ち手の発動条件と、相手側が取れる防衛策を併記せよ。
```

このアウトプットを Phase 3の多角レビューに加算し、ヘッジ戦略の十分性を評価する。

## 進行モード

依頼内容と「自走モード／確認モード」をPhase 0で1問だけ聞く:

```
中間チェックをどう運用しますか？
  A. Phase 2-A最速成果物のみ確認、それ以外は自走（推奨）
  B. 各Phaseで都度確認
  C. 確認ゼロ、最終提出まで完全自走
```
