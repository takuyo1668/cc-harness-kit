# フロントエンド作業時のルール

UI/フロントエンド変更を含む作業では以下を遵守する。

## 動作検証必須
- レイアウト・コンポーネント・状態を変える UI 変更は dev server を起動しブラウザで動作確認してから完了報告。文言修正・単一スタイル値のみの変更は差分提示＋確認ログで可
- 「動くはず」報告禁止。実行結果のスクリーンショット or 確認ログを提示
- レスポンシブ崩れ・文字重なり・余白の異常をチェック

## ビジュアル品質
- **HTML/デザイン成果物（Artifact公開・スライド・レポート等）は、ユーザーへ渡す前に必ず2段ゲートを通す（2026-07-01方針）**:
  1. **スクショ目視**: ヘッドレスChromeで実レンダリングを画像化しReadで自己確認（デスクトップ＋狭幅＋DOM実測でoverflow確認）。「レンダリングを見ずに公開」は禁止（過去に目次CSSカスケードで本文が210pxに潰れた事故あり）
  2. **design-review ゲート**: `design-review` スキル（ui-specialist / product-designer / tech-researcher の3名並列）を実行し、P0/P1を反映してから公開
- 上記の実行主体＝Artifact/成果物公開の直前手順。飛ばさない
- 画像・フォント追加時はパフォーマンス影響（LCP / CLS）を確認
- ピクセルパーフェクトを求められた場合はFigma変数・コンポーネントと差分確認

## アクセシビリティ
- alt属性・コントラスト比・キーボード操作・ARIA属性を確認
- フォーカスリングを消さない

## Next.js / React 固有
- Server Components / Client Components の境界を明示
- Hydration エラーが出ない構造（dateやrandomを直接renderしない）
- App Router の規約（page.tsx / layout.tsx / route.ts）に従う

## PDF 生成時
- pdftoppm で全ページ画像化 → Read で右端切れ・重なり・表記被り確認
- `@page` CSS 必須（A4 / 余白指定）
