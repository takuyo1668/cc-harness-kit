## 本質的原則（Boris Cherny / Claude Codeリードの運用哲学）

詳細ルールに入る前に、必ず内在化すべき5つの本質原則:

1. **Plan modeはデフォルト**: ほぼ全セッションをPlan modeから開始する。単純修正（差分を1文で説明できる・1ファイル変更・既知パターン）のみ直接着手。良いPlanが成果の大半を決める
2. **検証ループが品質を2-3倍にする**: Claudeに「結果を見る手段」を必ず与える（dev server / テスト / UI操作 / スクリーンショット / ログ）。**検証できないとClaudeは推測する**。推測完了報告は品質低下の最大原因
3. **知識はPRに埋め込む**: ミスしたら即CLAUDE.mdまたはfeedbackメモリに追記。gitリポジトリ運用時は `.claude/` 配下をgit管理してチーム共有
4. **繰り返し作業はslash command化・subagent化**: 同じ作業を2回手でやったらスキル化・コマンド化・subagent化する
5. **過度にカスタマイズしない**: Claude Codeは最初から完成度が高い。「これを消したらClaudeが間違いを犯すか?」がNOなルールは追加しない。少ない・濃いルールが最強

### ルール追加の原則（形骸化防止）
- 新しいルールには**実行主体**（settings.json deny / フック / スキル手順 / 定期レビュー項目）を必ず併記する
- 実行主体を持てない「心がけ」はルール化せず、feedbackメモリ止まりにする
- 同じ内容を複数ファイルに書かない。正本を1箇所決め、他は1行参照にする

### 関連ファイル

## 共通ルール

- git は使っていないプロジェクトもある。git 前提のワークフローを強制しない
- コード内コメントは{{日本語}}で書く
- テストファイルを勝手に変更しない（コードを直す。テスト変更が必要ならユーザー承認）
- 5ファイル以上の変更は事前申告
- 同一エラー修正は3回まで → 停止してユーザーにエスカレート
- `git add .` / `git add -A` 禁止。個別ファイル指定

## ワークフロー

- 設計判断を伴う3ステップ以上のタスクはプランモードで開始し、実装前にユーザーと合意する
  - ただしバグ修正・単一ファイル変更・既知パターンの実装は直接着手してよい
- リサーチ・探索・並列可能な作業はサブエージェントに委託して速度を上げる
- **思考タスクは着手前にFWを選ぶ**: 課題分解・原因分析・意思決定・優先順位付け・市場/競合分析・戦略立案・アイデア発想・目標設定・振り返り・提案/説明の構成など「考える／整理する／決める」作業では、いきなり書き始めず `logical-thinking` スキルの索引で最適な思考フレームワークを選んでから思考する（トークン最小・精度最大のため）
- バグ報告を受けたら質問せずに自律的に修正する。ログ・エラー・コードを自分で追う
- タスク完了前に必ず動作検証する
  - 実装・修正・設定変更タスクの**完了報告直前**に `verification-before-completion` スキルの言い訳封じ表を自己照合する（証拠なき完了報告＝推測完了報告を禁止。本質原則#2の実行主体）
- AskUserQuestion で確認するときは、質問本文に判断材料（元の状態・変更内容・影響）を完結に含める。直前のテキストに頼らない

### Generator-Evaluator 分離（ハーネス設計）
- **適用対象を限定**: 以下のタスクのみ生成と評価を分離する
  - 本番デプロイ前の成果物
  - 外部API連携・セキュリティに関わる変更
  - 新規アーキテクチャ・DB設計
- 上記以外のローカル変更・リファクタ・UI調整は分離不要。直接完了してよい
- 評価基準は事前にチェックリスト形式で定義する。「良い」「適切な」等の曖昧語は禁止し、検証可能な条件のみ使う
- 3回修正してもPASSしない場合はユーザーにエスカレートする

### コンテキストリセット
- 長時間タスク（7フェーズ以上）は各フェーズをサブエージェントに分離する
- 7フェーズ未満はcompactionに任せてよい
- 各エージェントは短いコンテキストで動作し、成果物のみを次に引き継ぐ

### ハーネス定期見直し（"What can I stop doing?"）
- 対象: 本ファイル / rules / commands / skills / agents / memory
- 自問: このルールはまだ必要か？　なしで品質が落ちた実績はあるか？　ボトルネックになっていないか？　日付・期限・プロジェクト名は現実と一致しているか？
- 不要と判断したルールは削除し、feedbackメモリに理由を記録する

## コード品質

- シンプルさ最優先。変更は必要最小限のコードだけに影響させる
- 一時的な回避策ではなく根本原因を修正する
- 非trivialな変更では「もっとシンプルな方法はないか」を一度確認してから実装する

## リスク管理

- 不可逆・破壊的操作（rm -rf / force push / DROP TABLE 等）の正本リストと確認手順は `~/.claude/rules/infrastructure.md`。機械ブロックは settings.json deny ＋ pretool-guard.sh の二層
- 外部API連携の変更時の確認事項（レート制限・認証・エラーハンドリング）は `~/.claude/rules/backend.md` が正本

## 自己学習

### 修正時の根本原因分類（パターン検出）
- ユーザーから修正を受けた場合、以下の原因カテゴリを特定して feedback メモリに含める:
  - **前提ミス**: 仕様・制約の確認不足
  - **過剰設計**: 不要な複雑さの導入
  - **検証不足**: 動作確認せず完了とした
  - **コンテキスト不足**: プロジェクト固有の事情を見落とした

### メタ認知ジャーナル（3行ふりかえり）
- **目的**: ユーザー自身のメタ認知力を鍛える。AIとの協業判断パターンを言語化・蓄積する
- **タイミング**: 実装タスク・意思決定を含む会話の終了時に、以下3行を最終レスポンスに含める
  - **判断**: この会話で下した中心的な意思決定は何か（1行）
  - **根拠**: なぜその判断にしたか。他にどんな選択肢があり、なぜ選ばなかったか（1-2行）
  - **次回の省略**: 次回同種タスクで省ける手順・自動化できる部分は何か（1行）
- **スキップ条件**: 雑談・単発質問・情報提供のみの会話では出力しない
- **頻度制御**: 1会話につき1回のみ
- **書き方**: ジャッジを入れず観察事実のみ。自己批判ではなく観察ログ

### 会話ログの保存（任意）
> 会話の現状（完了/未完了タスク・意思決定）を外部ノートに保存して次回再開できるようにする運用。
> 保存先（Obsidian等のノートアプリ／memory）は自分の環境に合わせて `docs/CUSTOMIZE.md` の手順で設定する。
